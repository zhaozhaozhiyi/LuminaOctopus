## 1. 背景与机会

许多用户需要将网站“下载到本地离线可浏览”，用于文档备份、帮助中心离线阅读、网站迁移前快照、资料归档等。参考产品 SiteSucker Pro 证明了这一类工具的长期需求，但其平台局限在 macOS。本项目目标是先提供 **Web 版可用的整站镜像工具**，再逐步演进为 **Tauri 2 桌面端（Windows + macOS）**，获得更好的本地保存与系统集成体验，同时尽量控制安装包体积。

## 2. 产品目标

- **G1 可用离线镜像**：对中小型站点形成可离线浏览的站点快照（HTML 入口与关键资源本地化）。
- **G2 可控与可恢复**：任务可暂停/恢复/停止，提供队列状态、错误与日志，可回看历史任务并导出结果。
- **G3 可演进**：爬虫引擎与 UI 解耦，后续 Tauri 桌面端可复用同一引擎；Web 版以 zip 导出为主，桌面版以落盘目录为主。

## 3. 非目标（本期明确不做）

- **NG1 不做浏览器级自动化爬虫**：不保证复杂 SPA、强交互 SaaS、无限滚动、复杂登录/MFA 的完整离线复现。
- **NG2 不做合规级取证归档**：不承诺 WARC、可证明的完整性链路等法证级能力。
- **NG3 不做全站“无限规模”抓取**：默认强调礼貌抓取与可控的深度/并发/延迟策略。

## 4. 目标用户与典型场景

### 4.1 目标用户

- **文档/知识工作者**：离线阅读帮助中心、教程、课程资料。
- **研发/运维/内容团队**：迁移前备份站点、回归对照、环境隔离下的文档镜像。
- **个人站长/设计师**：保存网站快照，留存视觉与内容。

### 4.2 典型使用场景（User Stories）

- **US1**：我输入一个文档站首页，设置抓取深度 2，点击开始，看到抓取进度与剩余队列，最终能离线打开首页并点击内部链接正常跳转。
- **US2**：抓取过程中我发现某个路径不需要，能对单条 URL 点击 Skip，避免浪费时间与带宽。
- **US3**：抓取中途我暂停任务，稍后恢复继续；即使刷新页面/服务重启，我也能从历史任务中找到该 job 并导出或继续。
- **US4**：当某些页面抓取失败，我能在日志中看到失败原因（状态码/错误信息/来源页面），便于调整设置重试。

## 5. 产品范围与阶段规划

本项目以“阶段可交付”为原则迭代。阶段的“完成”以 **验收标准** 为准（见各功能条目）。

### 5.1 Phase 1：核心抓取 MVP（已具备 / 稳定化）

**范围**
- URL 输入 + 校验与规范化
- 深度（Level/Depth）控制
- 开始 / 暂停 / 恢复 / 停止
- 队列列表（状态、URL/路径、进度、层级/类型）
- 汇总统计（已下载、剩余、错误、发现页面/资源等）
- 同站点限制、并发与延迟、robots 开关（至少可用）
- 任务落盘（状态 + 输出文件 + 日志）

**验收标准**
- 对中小型静态/半静态站点抓取成功率高，离线打开 `index.html` 可浏览主要页面与资源。
- 暂停/恢复/停止行为与 UI 状态一致，队列数与统计项可解释。
- 任务输出在 `jobs/<jobId>/site/` 可复用（导出/预览/恢复用）。

### 5.2 Phase 2：任务化体验（强烈建议优先完成）

**范围**
- 历史任务列表（Jobs）
- 选中任意 job 可查看其状态快照与日志
- 导出 zip：可导出 **选中 job** 的落盘站点文件
- 基础日志：按事件/级别展示（info/warn/error）

**关键产品要求（必须）**
- **状态查询按 jobId 生效**：切换历史 job 时 UI 展示必须可信。
- **导出按 jobId 生效且可跨重启**：刷新页面/重启服务后仍能导出历史 job。

**验收标准**
- Jobs 列表可见、可切换；每个 job 状态与日志一致。
- 导出 zip 可用且内容与 `siteRoot` 一致（含 HTML 与资源）。

### 5.3 Phase 3：队列可控与可预览（对齐参考产品关键体验）

**范围**
- 单条 Skip（跳过 URL，不再处理；可在列表中标记 skipped）
- 离线预览（Web）：对已抓取的 page 资源提供安全的预览入口
- 基础过滤与搜索：按状态筛选、按 URL 搜索、按类型（page/asset）筛选

**验收标准**
- Skip 生效：点击后队列剩余数下降或条目变为 skipped，且不会再被重新加入。
- 预览可用：对 page 条目可打开对应本地 HTML（经改写），页面资源引用正常。

### 5.4 Phase 4：设置系统与抓取规则增强（中长期）

**范围**
- 设置面板完善：并发、延迟、同站点策略、User-Agent、重试/超时、大小限制、最大文件数
- URL include/exclude（字符串或正则）
- 发现增强：扫描 sitemap、扫描注释、支持自定义 data-* 属性扫描（参考 SiteSucker）
- 资源类型过滤（按 MIME/扩展名策略）
- 更完善 robots 策略（含 meta robots / x-robots-tag 的处理边界说明）

**验收标准**
- 在常见站点场景下，用户无需改代码即可通过设置显著改善抓取覆盖率或稳定性。

### 5.5 Phase 5：Tauri 2 桌面端（规划）

**范围**
- 选择保存目录（系统对话框）
- 一键在 Finder/资源管理器中打开结果目录
- 运行在本地，减少对“Next API 作为爬虫宿主”的依赖（桌面端本地进程运行引擎）
- 前端桥接同步状态与日志（复用现有 Next.js/React UI，桌面端以静态资源加载为主）
- 不做自动更新（安装包分发与手动升级）

**验收标准**
- 桌面端抓取与 Web 版核心能力一致；导出从“zip”转向“落盘目录”为主，体验更自然。

## 6. 功能需求（当前形态 + 详细要求）

### 6.1 抓取会话（Crawl Session）

**输入项**
- **入口 URL**：必填，支持 `http/https`，自动 trim、规范化。
- **最大深度**：0~5（默认 2）。
- **并发数**：1~6（默认 3）。
- **请求延迟**：0~5000ms（默认 200ms）。
- **仅同站点页面**：默认开启（sameSitePagesOnly）。
- **遵守 robots**：默认开启（respectRobots）。

**行为**
- 点击开始：创建新 jobId，状态进入 running。
- 暂停：状态进入 paused，保留队列与 in-flight 信息。
- 恢复：paused → running。
- 停止：running/paused → stopped；可选择稍后继续（若队列仍有剩余）。

**展示**
- 汇总区：depth、已下载、剩余、错误、发现页面数、资源数、降级页面数（degradedPages）等。
- 列表：`Status | URL/Path | Progress | (Skip/Preview)`，并显示 `kind(page/asset)` 与 `level`。

**验收标准**
- 同一 URL 不重复入队（hash 去除后去重）。
- 页面抓取：渲染/改写后落盘，并能离线引用资源。
- 资源抓取：CSS 做 url() 改写；二进制资源按 Buffer 保存。

### 6.2 Jobs（任务/历史）

**功能**
- 展示历史任务列表（jobId、baseUrl、状态、开始/结束时间、下载数、错误数、warnings）。
- 选择 job：切换详情视图（状态快照、队列 items、日志）。
- 活跃任务提示：对 running/paused 任务给予显著标识。

**验收标准**
- 切换 job 后，状态/日志与该 job 的落盘内容一致。

### 6.3 Logs（日志）

**功能**
- 展示最近 N 条日志（默认 80~200），按时间排序。
- 支持级别筛选（info/warn/error）。
- 关键事件建议（用于后续可视化与排障）：
  - `job_started/job_stopped/job_paused/job_resumed`
  - `page_saved/asset_saved`
  - `request_failed/parse_failed/rewrite_degraded`
  - `skipped`

**验收标准**
- 失败时日志能定位 URL + 错误原因（状态码/异常 message）。

### 6.4 Export（导出）

**功能**
- 导出 zip：可导出选中的 job（jobId）在 `siteRoot` 下的全部文件（或按 manifest）。
- 生成文件名：包含时间戳与 baseUrl（可选，需安全处理）。

**验收标准**
- **跨刷新/重启可导出**：即使当前无内存引擎，也能从 job 目录导出。

### 6.5 Skip（单条跳过）

**功能**
- 对队列条目支持 Skip：
  - 未开始的队列项：直接移出队列并标记 skipped
  - 已完成/失败项：允许标记（用于 UI 隐藏或归类），但不影响结果
  - 正在进行的项：默认不强制中断（可作为后续增强），但应阻止其后续再次入队

**验收标准**
- Skip 后，该 URL 不再被下载/写入；UI 统计与队列剩余数变化符合预期。

### 6.6 Preview（离线预览）

**功能**
- 对已抓取的 `page` 资源提供预览入口（新 tab 或内置 iframe）。
- 安全约束：
  - 仅允许访问 `jobs/<jobId>/site/` 目录内文件
  - 防止目录穿越（`../`）与读取 workspace 其他文件

**验收标准**
- 预览页面的相对资源路径可正确加载，至少在同域资源场景下表现良好。

## 7. 信息架构与界面（IA / UI）

### 7.1 IA

Lumina Octopus
- 主界面：抓取控制 + 汇总 + 队列列表
- 历史（Jobs）：列表 + 选择 job
- 日志（Logs）：与 job 关联的时间线
- 设置（Settings）：抓取参数与规则（逐步完善）
- 导出（Export）：下载 zip（Web）/ 打开目录（桌面端）

### 7.2 关键交互规则

- 运行中的任务应有明确的“当前 active job”概念：暂停/恢复/停止默认作用于 active job。
- 历史 job 主要是回看/导出/预览；恢复继续应在 UI 上清晰提示“会启动引擎并占用 active job”。

## 8. 数据与接口（产品视角）

### 8.1 核心数据模型（抽象）

- **Job**：`jobId`、`baseUrl`、`status`、`startedAt/stoppedAt`、统计项、warnings、lastError
- **Item**：`url`、`status`、`progress`、`level`、`kind(page/asset)`、`localPath`、`contentType`、`error`、`updatedAt`
- **LogEntry**：`timestamp`、`level`、`event`、`message`、`url?`

### 8.2 API（当前/规划）

- `POST /api/crawl/start`：创建 job 并启动
- `GET /api/crawl/jobs`：历史 job 列表
- `GET /api/crawl/status?jobId=...`：读取指定 job 的状态（**必须按 jobId 生效**）
- `GET /api/crawl/logs?jobId=...&limit=...`：读取日志
- `POST /api/crawl/pause` / `POST /api/crawl/resume` / `POST /api/crawl/stop`：控制 active job（中期可扩展为按 jobId）
- `GET /api/crawl/export?jobId=...`：导出指定 job（**必须从落盘导出**）
- `POST /api/crawl/skip`：跳过 URL（规划）

## 9. 非功能需求（NFR）

### 9.1 性能

- UI 列表在 1k+ items 时仍可操作（需要分页/虚拟列表或后端分页）。
- 轮询频率可控：状态与日志轮询不应导致高 IO/高 CPU（服务端应节流写盘，客户端可降低无变化时的频率）。

### 9.2 稳定性与可恢复

- 任务状态与输出落盘；进程重启后历史可回看、可导出。
- 失败不应阻塞整体：单条资源失败可记录并继续。

### 9.3 安全

- 预览与导出必须严格限制文件访问范围，禁止任意文件读取。
- URL 处理与重写需避免注入（仅作为内容处理，不执行不可信脚本）。

### 9.4 合规与伦理提示（产品文案要求）

- 对 robots、跨域资源、版权内容应给出提示：工具用于备份/离线阅读，请用户自行确保有权抓取与使用。

## 10. 风险与对策

- **R1 动态站点抓取不完整**：明确边界；提供“渲染模式/等待时间”等设置与建议。
- **R2 规模过大导致资源耗尽**：提供最大深度/最大文件数/大小限制；在 UI 提示风险。
- **R3 导出/预览的安全风险**：严格路径白名单与输入校验；必要时只允许按 manifest 导出。
- **R4 多 job 并行与引擎单例冲突**：短期只支持单 active job；中期引擎与 jobId 绑定并支持队列化。

## 11. 里程碑（建议）

- **M1（1-3 天）**：状态查询按 jobId 生效；导出按 jobId 从落盘导出；回归测试“刷新/重启后导出”。
- **M2（3-5 天）**：Skip（API+引擎+UI）；列表筛选/搜索。
- **M3（5-8 天）**：离线预览（安全静态路由）；预览按钮；基础设置面板补齐。
- **M4（2-4 周）**：设置系统、规则增强（include/exclude、sitemap、类型过滤等）。
- **M5（后续）**：Tauri 2 桌面端封装与本地目录体验。
